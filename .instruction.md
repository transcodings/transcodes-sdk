# Rollup.js & Changesets Guide

## 1. What is Rollup.js?

Rollup is a **bundler specialized for publishing npm libraries**.
While webpack excels at application bundling, Rollup is used for building library packages.

### Core Concepts

```
src/ (TypeScript source)
    ↓ rollup bundling
lib/
  ├── index.cjs      ← CommonJS (Node.js, require)
  └── index.esm.js   ← ESM (import/export, Tree Shaking)
```

### Basic Configuration (rollup.config.js)

```javascript
import typescript from 'rollup-plugin-typescript2';
import del from 'rollup-plugin-delete';

export default {
  input: 'src/index.ts', // Entry point
  output: [
    {
      file: 'lib/index.cjs',
      format: 'cjs', // require() style
      exports: 'named',
    },
    {
      file: 'lib/index.esm.js',
      format: 'esm', // import style (supports Tree Shaking)
      exports: 'named',
    },
  ],
  plugins: [
    del({ targets: ['lib/*'] }), // Clean lib/ before build
    typescript({ useTsconfigDeclarationDir: true }), // TS → JS compilation
  ],
};
```

### When External Libraries Are Involved

```javascript
import resolve from '@rollup/plugin-node-resolve';
import commonjs from '@rollup/plugin-commonjs';

const external = [...Object.keys(packageJson.peerDependencies || {})];

export default {
  plugins: [
    resolve(), // Resolve packages from node_modules
    commonjs(), // Convert CJS packages to ESM
  ],
  external, // Exclude peerDependencies from the bundle
};
```

| Plugin                        | Purpose                              |
| ----------------------------- | ------------------------------------ |
| `rollup-plugin-delete`        | Delete `lib/` folder before build    |
| `rollup-plugin-typescript2`   | TypeScript compilation + `.d.ts` generation |
| `@rollup/plugin-node-resolve` | Resolve `node_modules` packages      |
| `@rollup/plugin-commonjs`     | CJS → ESM conversion                |

### package.json Configuration

```json
{
  "main": "lib/index.cjs",
  "module": "lib/index.esm.js",
  "types": "lib/index.d.ts",
  "sideEffects": false,
  "publishConfig": {
    "access": "public"
  },
  "scripts": {
    "build": "rollup -c",
    "release-package": "changeset publish"
  }
}
```

## 2. What is Changesets?

Changesets is a **version management + automated CHANGELOG generation + npm publish automation tool**.

```
Manual:
  Edit package.json version → Write CHANGELOG → npm publish

Changesets:
  npx changeset → Changesets handles the rest automatically
```

---

## 3. Initial Setup (init)

### Installation

```bash
npm install -D @changesets/cli
```

### Initialization

```bash
npx changeset init
```

Generated structure:

```
.changeset/
  └── config.json    ← Changesets configuration file
```

### config.json Configuration

```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

## 4. Standard Release Workflow

### Step 1: Record Changes

```bash
npx changeset
```

Follow the terminal prompts in order:

```
1. Select packages (use spacebar to select)
2. Select change type:
   - major: 1.0.0 → 2.0.0  (Breaking Change)
   - minor: 1.0.0 → 1.1.0  (New feature)
   - patch: 1.0.0 → 1.0.1  (Bug fix)
3. Enter change description
```

A file is created in the `.changeset/` folder:

```
.changeset/
  └── fuzzy-bears-dance.md   ← Auto-generated filename
```

### Step 2: Update Version

```bash
npx changeset version
```

This automatically:

- Updates `package.json` version
- Creates/updates `CHANGELOG.md`
- Deletes `.changeset/` files

### Step 3: Build + Publish

```bash
npm run build
npx changeset publish
```

## 5. Pre-release Mode (alpha / beta)

### Starting Alpha

```bash
npx changeset pre enter alpha
```

From this point, versions will follow this pattern:

```
1.0.0 → 1.0.1-alpha.0
```

### Recording Alpha Changes + Publishing

```bash
# Record changes
npx changeset

# Update version
npx changeset version
# → 1.0.1-alpha.1, 1.0.1-alpha.2 ...

# Build + publish
npm run build
npx changeset publish
```

### Transitioning from Alpha → Beta

```bash
# Exit alpha
npx changeset pre exit

# Enter beta
npx changeset pre enter beta
```

Version progression:

```
1.0.1-alpha.2 → 1.0.1-beta.0
```

### Transitioning from Beta → Stable Release

```bash
# Exit beta
npx changeset pre exit

# Update version
npx changeset version
# → 1.0.1 (stable version)

# Build + publish
npm run build
npx changeset publish
```

## 6. Full Pre-release Flow Summary

```
Start development
    ↓
npx changeset pre enter alpha   → Enter alpha mode
    ↓
Code changes → npx changeset    → Record changes
    ↓
npx changeset version           → 1.0.1-alpha.0
    ↓
npx changeset publish           → Publish to npm
    ↓
(repeat)                        → 1.0.1-alpha.1, alpha.2 ...
    ↓
npx changeset pre exit          → Exit alpha
npx changeset pre enter beta    → Enter beta mode
    ↓
npx changeset version           → 1.0.1-beta.0
    ↓
npx changeset publish           → Publish to npm
    ↓
(repeat)                        → 1.0.1-beta.1, beta.2 ...
    ↓
npx changeset pre exit          → Exit beta
    ↓
npx changeset version           → 1.0.1 (stable)
    ↓
npx changeset publish           → Stable release complete
```

## 7. GitHub Actions Automation

```yaml
name: Release

on:
  push:
    branches:
      - main
      - prerelease

concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    name: Release
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0
          token: ${{ secrets.GIT_CHANGESET_TOKEN }}

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build

      - name: Create release pull request or publish
        uses: changesets/action@v1
        with:
          publish: npm run release-package
        env:
          GITHUB_TOKEN: ${{ secrets.GIT_CHANGESET_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_CHANGESET_TOKEN }}
```

### How It Works

```
Push to main branch
    ↓
If .changeset/ files exist
    → Automatically creates "Version Packages" PR

When "Version Packages" PR is merged
    → Automatically updates version
    → Automatically publishes to npm
    → Automatically creates GitHub Release
```
